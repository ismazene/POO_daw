01

Iterator con ArrayList

Aparece en: ★★★ todos los exámenes

¿Por qué usar Iterator para eliminar?

Si intentas eliminar elementos de una lista mientras la recorres con for-each, Java lanza una ConcurrentModificationException. El Iterator tiene su propio método remove() que actualiza el cursor interno de forma segura.

Trampa del examen: ¿por qué no se puede usar for-each? → Porque no expone un método remove() seguro. Si llamas a lista.remove() dentro del for-each, la lista se modifica mientras se itera → ConcurrentModificationException.

Patrón completo — eliminar elementos que cumplen condición

ArrayList<Integer> lista = new ArrayList<>(Arrays.asList(2, 4, 6, 8, 10));
Iterator<Integer> it = lista.iterator();

while (it.hasNext()) {
    int num = it.next();
    if (num > 5) {          // condición que cambia en cada examen
        it.remove();        // SIEMPRE it.remove(), nunca lista.remove()
    }
}
// lista resultante: [2, 4]

En el examen de abril la condición era > 5, en mayo era < 8. La estructura siempre es idéntica.

02

Polimorfismo

Aparece en: ★★★ todos los exámenes

Ligadura dinámica (late binding)

La pregunta siempre tiene este patrón: referencia de tipo padre, objeto de tipo hijo. Java decide qué método ejecutar en tiempo de ejecución, según el tipo real del objeto (no el de la referencia).

Animal a = new Perro();   // referencia Animal, objeto Perro
a.hacerSonido();           // imprime: "¡Guau!" → método de Perro

Se ejecuta el de Perro aunque la referencia sea de tipo Animal. Esto es la ligadura dinámica.

Segunda parte — si el método NO está en la clase hija

Si hacerSonido() no existe en Perro, Java sube por la jerarquía de herencia y ejecuta el de Animal: "El animal está haciendo un sonido!".

Java siempre busca el método de abajo hacia arriba en la jerarquía.

En el examen de mayo fue con Instrumento / Guitarra y el método tocar(). La mecánica es exactamente la misma. Identifica el patrón.

Ejemplo completo del examen

class Animal {
    public void hacerSonido() {
        System.out.println("El animal está haciendo un sonido!");
    }
}
class Perro extends Animal {
    public void hacerSonido() {
        System.out.println("¡Guau!");
    }
}

Animal a = new Perro();
a.hacerSonido();  // → "¡Guau!"  (objeto es Perro, se ejecuta Perro)

Encapsulación

Aparece en: ★★★ todos los exámenes — con la misma clase Persona

El problema: atributo público sin control

public class Persona {
    public int edad;   // ← cualquiera puede poner p.edad = -5
}

Al ser public, cualquier clase externa puede asignar un valor inválido sin ninguna validación. Esto rompe la integridad del objeto.

La solución: private + getter + setter con validación

public class Persona {
    private int edad;  // 1. private → nadie accede directamente

    // 2. getter: devuelve el valor
    public int getEdad() {
        return edad;
    }

    // 3. setter: valida antes de asignar
    public void setEdad(int edad) {
        if (edad >= 0) {
            this.edad = edad;
        } else {
            System.out.println("Error: edad no puede ser negativa");
            // o: throw new IllegalArgumentException("Edad inválida");
        }
    }
}

Ahora p.edad = -5 no compila. El acceso es solo a través del setter, que valida.

Los tres conceptos clave a mencionar en el examen: private, getter, setter con validación.

Recursividad

Aparece en: ★★★ la profe la enfatizó especialmente

Los dos elementos que siempre tiene una función recursiva

1. Caso base Condición que detiene la recursión. Sin él → bucle infinito → StackOverflowError.

2. Llamada recursiva El método se llama a sí mismo con un argumento que se acerca al caso base.

Ejemplo del examen de mayo — recursiva(4)

public static void recursiva(int n) {
    if (n == 0) return;   // ← caso base
    System.out.print("<");
    recursiva(n - 1);      // ← llamada recursiva ANTES de ">"
    System.out.print(">");
}

Ejemplo clásico — potencia(2, 4)

static int potencia(int base, int exponente) {
    if (exponente == 0) return 1;          // caso base: b^0 = 1
    return base * potencia(base, exponente - 1);
}

Si el exponente fuera negativo: nunca se alcanza el caso base (exponente == 0), la recursión no termina → StackOverflowError. Solución: añadir validación al inicio.

Método infalible: una tabla con una columna por variable

Cada fila = una iteración del bucle interior. Rellena celda por celda, sin saltar pasos.

Excepciones personalizadas

Aparece en: ★★ exámenes de abril y mayo — mismo ejercicio

Estructura de una excepción personalizada

// 1. Crear la clase que extiende Exception (o RuntimeException)
public class FondosInsuficientesException extends Exception {
    public FondosInsuficientesException(String mensaje) {
        super(mensaje);   // pasa el mensaje a la clase padre
    }
}

// 2. Usarla con throw + declarar throws en la firma
public void retirar(double cantidad, double saldo)
        throws FondosInsuficientesException {
    if (cantidad > saldo) {
        throw new FondosInsuficientesException(
            "Fondos insuficientes. Saldo: " + saldo);
    } else {
        saldo -= cantidad;
    }
}

¿Exception o RuntimeException? — la justificación que piden

TipoCuándo usarla¿El compilador obliga a capturarla?
Exception (checked)Errores recuperables: fondos insuficientes, archivo no encontradoSí → hay que poner try/catch o throws
RuntimeException (unchecked)Errores de programación: índice inválido, null pointerNo → es opcional capturarla

Para el sistema bancario, Exception tiene más sentido: el usuario puede intentar retirar una cantidad menor, es un error recuperable y vale la pena obligar al programador a manejarlo.

Interfaces y herencia múltiple

Aparece en: ★★ exámenes de abril, mayo y junio

¿Cómo resuelve Java la herencia múltiple?

Java no permite heredar de dos clases a la vez (extends ClaseA, ClaseB no compila). La solución son las interfaces: una clase puede implementar cuantas quiera.

interface Volador  { void volar();  }
interface Nadador  { void nadar();  }

class Pato extends Animal
        implements Volador, Nadador {   // múltiples interfaces
    public void volar() { System.out.println("Volando"); }
    public void nadar() { System.out.println("Nadando"); }
}

Clase anónima — solución temporal (examen de junio)

Si no existe ninguna clase que implemente una interfaz pero necesitas usarla puntualmente, puedes crear una clase anónima directamente en el código:

GestionInventario g = new GestionInventario() {
    public void verificarStock() {
        System.out.println("Verificando stock temporalmente...");
    }
};
g.verificarStock();

Esto crea una clase sin nombre que implementa la interfaz al vuelo, sin necesidad de crear un archivo .java nuevo.

Arrays bidimensionales

Aparece en: ★★ exámenes de abril y mayo

¿Qué imprime este código?

int[][] matriz = {
    {5, 3, 2},
    {1, 4, 6},
    {7, 8, 9}
};
int suma = 0, j = 1;     // j fijo = columna 1
for (int i = 0; i < matriz.length; i++) {
    suma += matriz[i][j]; // suma columna 1: 3 + 4 + 8
}
System.out.println(suma);
ij (fijo=1)matriz[i][j]suma
01matriz[0][1] = 30+3 = 3
11matriz[1][1] = 43+4 = 7
21matriz[2][1] = 87+8 = 15
15

Sumar siempre la última fila — sin importar tamaño

int suma = 0;
int ultimaFila = matriz.length - 1;  // índice de la última fila
for (int j = 0; j < matriz[ultimaFila].length; j++) {
    suma += matriz[ultimaFila][j];    // 7 + 8 + 9 = 24
}
System.out.println(suma);  // 24

Clave: matriz.length = número de filas. matriz[i].length = número de columnas de la fila i.

Lombok — @AllArgsConstructor, @Getter

Aparece en: ★ examen de junio (convocatoria extraordinaria)

Qué hace cada anotación

AnotaciónGenera automáticamente
@AllArgsConstructorConstructor con todos los atributos como parámetros
@NoArgsConstructorConstructor vacío (sin parámetros)
@GetterGetters para todos los atributos privados
@SetterSetters para todos los atributos privados
@Data@Getter + @Setter + toString + equals + hashCode

Ejemplo del examen de junio

@AllArgsConstructor   // genera: new Alumno(nia, nombre)
@Getter               // genera: getNia(), getNombre()
class Alumno {
    private int nia;
    private String nombre;

    public static void obtenerDescripcion(Alumno alumno) {
        System.out.println("Alumno: " + alumno.getNombre()
            + ", NIA: " + alumno.getNia());
    }
}

// Llamada desde main:
Alumno a = new Alumno(12345, "Ana");   // @AllArgsConstructor
Alumno.obtenerDescripcion(a);             // método estático → Alumno.metodo()

¿Podemos añadir @NoArgsConstructor? Con Lombok sí, pero hay que tener cuidado: si tienes atributos final, el constructor vacío no puede dejarlos sin valor → error. En este caso los atributos no son final, así que sería posible.
